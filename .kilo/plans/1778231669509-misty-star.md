# Implementation Plan: quota-reset combo + quota refresh-on-error + prompt-cache fast path

## Scope
Continue the quota-reset implementation and add prompt-cache/read-cache optimizations so the new routing path stays fast and does not add avoidable per-request CPU/DB overhead.

## Current state / constraints
- Source-of-truth plan already requested `quota-reset`, cached quota summaries, read-only auth inspection, async quota refresh on suspicious provider errors, UI/schema updates, and tests.
- `open-sse/services/AGENTS.md` requires no blocking/network I/O in combo hot paths. Any live quota fetch must remain async fire-and-forget outside routing.
- There are already partial working-tree changes for:
  - `getConnectionQuotaSummary()` in `quotaCache.ts`
  - `inspectProviderConnectionsForQuotaReset()` in `auth.ts`
  - `quotaRefreshOnError.ts`
  - `quota-reset` constants/UI fallbacks
  - combo branch for quota-reset ordering
- Plan should finish and harden those changes, not restart with a broad refactor.

## 1. Finish quota-reset routing correctness
1. Verify `sortTargetsByQuotaReset()` in `open-sse/services/combo.ts`:
   - keep original order as tie-breaker;
   - skip targets whose inspected account candidates are all unavailable/quota-exhausted/quota-policy-blocked;
   - if every target is skipped, return original list so existing `isModelAvailable()` / preflight returns the normal unavailable/all-rate-limited response;
   - do not call live quota APIs.
2. Avoid circular/import fragility:
   - keep `auth.ts` import lazy inside the quota-reset branch;
   - if typecheck reports circular module issues, move only the pure quota ranking/inspection primitives to a neutral domain/lib module, leaving DB access in `auth.ts`.
3. Ensure provider key is correct:
   - use `target.providerId || target.provider` for connection lookup;
   - use parsed requested model from `target.modelStr` for model lock / per-account exclusion checks.

## 2. Harden auth quota inspection helper
1. Ensure `inspectProviderConnectionsForQuotaReset(provider, allowedConnections, requestedModel, options)` mirrors credential selection filters:
   - active provider aliases;
   - `allowedConnections`;
   - forced `connectionId`;
   - terminal statuses;
   - `rateLimitedUntil`;
   - Codex scope lockouts;
   - per-model lockouts;
   - per-account model exclusions;
   - quota policy via `evaluateQuotaLimitPolicy()`;
   - cached quota exhaustion via `getConnectionQuotaSummary()` / `isAccountQuotaExhausted()` semantics.
2. It must not update `lastUsedAt`, `consecutiveUseCount`, backoff state, or call `preflightQuota()`.
3. Return enough metadata for deterministic ranking: `available`, `quotaExhausted`, `quotaBlocked`, `nearestResetAt`, `quotaHeadroomPercent`, `priority`, `lastUsedAt`, `backoffLevel`.

## 3. Add a fast read cache for quota-reset inspection
Goal: keep quota-reset from doing repeated DB reads when a combo has many targets for the same provider.

1. Add a small request-local memo inside `sortTargetsByQuotaReset()`:
   - key: `provider|allowedConnectionIds|requestedModel|forcedConnectionId`;
   - value: promise/result from `inspectProviderConnectionsForQuotaReset()`;
   - this removes duplicate DB reads inside one combo routing decision.
2. Consider a short process-level TTL cache only if profiling/typecheck/tests indicate hot-path pressure remains:
   - TTL: 1–3 seconds, max entries ~200;
   - key same as above;
   - never cache live quota fetch results, only inspection output derived from local DB + quota cache;
   - invalidate naturally via TTL, no complex invalidation required.
3. Prefer request-local memo first because it has no stale cross-request behavior and is enough for combos with repeated provider targets.

## 4. Prompt-cache optimization / “speed up all that shit”
There are two relevant cache paths:
- provider-side prompt caching (`prompt_cache_key`, Claude `cache_control`);
- local in-memory prompt/response LRU (`src/lib/cacheLayer.ts`), mostly separate from quota routing.

Implementation plan:
1. Make `quota-reset` cache-control-safe where possible:
   - update `open-sse/utils/cacheControlPolicy.ts` so `quota-reset` is treated as deterministic for cache-control preservation only when the final selected provider supports caching;
   - since `shouldPreserveCacheControl()` is called after target provider is known, adding `quota-reset` to `DETERMINISTIC_STRATEGIES` is acceptable for provider-side cache headers, but note it may rotate accounts; provider cache hit rate may be account-scoped.
2. Avoid expensive repeated prompt-cache-key generation:
   - inspect `src/lib/promptCache/prefixAnalyzer.ts` and `chatCore.ts` import at the injection point;
   - add a lightweight request-local cache in `chatCore.ts` around `generatePromptCacheKey(bodyToSend.messages)` keyed by the `messages` object reference or a stable prefix hash;
   - do not introduce global caching of raw prompts/messages unless bounded and redacted because prompts may contain sensitive data.
3. Improve `generatePromptCacheKey()` stability/performance if needed:
   - include role + content in the prefix hash, not only content, to avoid accidental collisions across role changes;
   - keep hashing only the detected prefix, not the full request, so large trailing conversation history does not cost extra;
   - optionally memoize `analyzePrefix(messages)` in a `WeakMap<Message[], PrefixAnalysis>` for the lifetime of the process; because keys are object references and weak, it should not retain prompts.
4. Do not enable local response caching for streaming/non-idempotent chat as part of this change. That is a separate behavior change with correctness/privacy implications.
5. Add tests:
   - `cache-control-policy.test.ts`: `quota-reset` preserves cache-control for caching-aware clients + cacheable provider, respects `never` override;
   - `promptCache` tests: role changes affect generated key; repeated analysis can be called safely and returns stable key.

## 5. Finish quota refresh-on-error
1. Keep scheduler fire-and-forget:
   - trigger on 429, 402, quota/rate-limit/usage-limit/exhausted text;
   - conservative for 403: only quota/credits text;
   - skip 401/auth/banned cases unless classifier says quota-like.
2. Throttle per `connectionId` (60–120s) and cap concurrency (2–3).
3. Extend `SyncSource` to include `"error"` and make `fetchAndPersistProviderLimits(connectionId, "error")` not force refresh unless desired.
4. Hook points:
   - `markAccountUnavailable()` after error classification;
   - optionally `chat.ts` near `markAccountExhaustedFrom429()` for exact connectionId, but avoid double scheduling if `markAccountUnavailable()` already covers the same error.
5. Unsupported providers must log debug and never break request flow.

## 6. Fix likely compile issues from partial changes
1. `src/types/combo.ts` currently has an `import type` inserted after declarations; move imports to top-level if typecheck complains.
2. Format long lines introduced in `auth.ts`, `combo.ts`, and `quotaRefreshOnError.ts`.
3. Ensure `setTimeout(...).unref?.()` is accepted in TS target; if not, store timer then `timer.unref?.()`.
4. Ensure `ProviderLimitsCacheEntry.source` type accepts `"error"`; if it is declared elsewhere in `localDb` types, update that union too.
5. Ensure `ACCOUNT_FALLBACK_STRATEGY_VALUES` should or should not include `quota-reset`:
   - plan target is combo strategy, not necessarily account fallback strategy;
   - leave account fallback unchanged unless UI/settings require it.

## 7. Tests to add/update
1. `tests/unit/sse-auth.test.ts`
   - inspection returns available known-reset candidate before unknown quota;
   - exhausted candidate is unavailable/skipped;
   - past reset is treated available via quota cache semantics;
   - quota policy threshold blocks candidate.
2. `tests/unit/combo-routing-engine.test.ts`
   - `quota-reset` tries target with nearest future reset first;
   - exhausted-only target is not attempted when another eligible target exists;
   - fallback continues if first eligible target returns an error.
3. `tests/unit/settings-schema-routing-strategies.test.ts`
   - schema accepts `quota-reset`.
4. `tests/unit/cache-control-policy.test.ts`
   - cache-control preservation includes `quota-reset` for caching providers and respects overrides.
5. New or existing quota refresh tests:
   - repeated 429 schedules one refresh due throttle;
   - unsupported provider failure is swallowed;
   - successful fetch path updates quota cache via existing provider limits flow.
