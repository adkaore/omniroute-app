# План: quota-reset-aware combo + автообновление квот при ошибках

## Цель
Добавить стратегию combo, которая ранжирует цели так, чтобы первой шла учётная запись/цель с доступной квотой и ближайшим будущим `resetAt` лимита. При этом аккаунты с уже исчерпанной квотой не должны выбираться для реального запроса, чтобы не ждать cooldown. Дополнительно — при ошибках провайдера автоматически перепроверять квоту аккаунта и обновлять quota cache/снапшоты, чтобы роутинг быстрее узнавал о фактическом исчерпании/восстановлении.

## Что уже есть в проекте
- Основной combo engine: `open-sse/services/combo.ts`.
  - Поддерживает стратегии `priority`, `weighted`, `round-robin`, `fill-first`, `p2c`, `least-used`, `cost-optimized`, `strict-random`, `auto`, `lkgp`, `context-optimized`, `context-relay`.
  - Для combo перед попыткой вызывается `isModelAvailable`, который через `getProviderCredentialsWithQuotaPreflight()` уже умеет отфильтровывать аккаунты без доступной квоты.
- Выбор аккаунта провайдера: `src/sse/services/auth.ts`.
  - Уже есть quota-aware фильтр: `isAccountQuotaExhausted()` исключает исчерпанные аккаунты из приоритета.
  - `evaluateQuotaLimitPolicy()` блокирует аккаунты, достигшие настроенного порога лимита.
  - `p2c` уже учитывает quota headroom, но не сортирует по ближайшему reset.
- Кэш квот: `src/domain/quotaCache.ts`.
  - Хранит `remainingPercentage`, `resetAt`, `exhausted`, `nextResetAt`.
  - Фоново обновляет известные entries раз в 5 минут.
  - `isAccountQuotaExhausted()` оптимистично разблокирует запись, если `resetAt` уже прошёл.
- Live fetch квот: `src/lib/usage/providerLimits.ts`.
  - `fetchAndPersistProviderLimits(connectionId, source)` получает usage, пишет provider limits cache, а через `setQuotaCache()` обновляет quota cache и `quota_snapshots`.
- UI/валидация стратегий:
  - Список стратегий: `src/shared/constants/routingStrategies.ts`.
  - Zod schemas используют `ROUTING_STRATEGY_VALUES` в `src/shared/validation/schemas.ts`.
  - Dashboard combos UI строит список из `ROUTING_STRATEGIES` в `src/app/(dashboard)/dashboard/combos/page.tsx`.

## Предлагаемое решение

### 1. Новая стратегия combo: `quota-reset`
Семантика: “сначала аккаунт/цель с доступной квотой и ближайшим будущим reset лимита”. Это ближе к идее «аккаунт, который ближе всего к перезапуску лимита», но безопасно: исчерпанные аккаунты не должны попадать в реальные попытки.

Поведение:
1. Собрать executable targets обычным `resolveComboTargets()`.
2. Применить tag routing, как сейчас.
3. Для каждой цели определить её quota state:
   - если `target.connectionId` есть: смотреть cache этой конкретной учётки;
   - если `target.connectionId` нет, но provider имеет несколько аккаунтов: использовать helper выбора/инспекции аккаунтов для provider, чтобы понять лучшую доступную учётку без реального запроса;
   - если quota данных нет: цель остаётся eligible, но ранжируется после целей с известным ближайшим reset.
4. Исключить/отложить вниз цели, где аккаунт явно исчерпан (`isAccountQuotaExhausted(connectionId) === true`) или quota policy reached threshold.
5. Отсортировать оставшиеся:
   - сначала доступные с future `resetAt`, по возрастанию `resetAt`;
   - затем доступные без quota данных;
   - затем targets без конкретной учётки/неподдерживаемые quota providers по исходному порядку;
   - исчерпанные лучше полностью не включать в ordered list, чтобы `handleComboChat()` их не пробовал. Если все исчерпаны — вернуть 503/429-like unavailable с ближайшим reset, либо оставить текущий preflight путь, который вернёт `allRateLimited`.

Почему не использовать существующий `p2c`:
- `p2c` оптимизирует по health/headroom/penalty и случайной выборке.
- Здесь нужна детерминированная сортировка по ближайшему reset при наличии доступной квоты.

### 2. Account-level helper для quota-reset выбора
В `src/sse/services/auth.ts` стоит добавить экспортируемый helper, чтобы не дублировать фильтры:
- `inspectProviderConnectionsForQuotaReset(provider, allowedConnections, requestedModel, options)` или похожее имя.
- Он должен переиспользовать существующую логику нормализации connections, фильтра unavailable/terminal/model lock/quota policy.
- Возвращать список кандидатов с полями:
  - `connectionId`
  - `provider`
  - `available: boolean`
  - `quotaExhausted: boolean`
  - `quotaBlocked: boolean`
  - `nearestResetAt: string | null`
  - `quotaHeadroomPercent: number | null`
  - `priority`, `lastUsedAt`, `backoffLevel`
- Для forced target (`target.connectionId`) можно напрямую оценить его состояние.

Важно: helper не должен делать live fetch на hot path. Он должен читать DB/cached quota state, чтобы не тормозить routing. Live fetch — отдельный механизм на ошибках/фоновой синхронизации.

### 3. Расширить `quotaCache` публичными методами
Добавить безопасные helper-функции:
- `getConnectionQuotaSummary(connectionId)`:
  - `exhausted`
  - `nearestResetAt` среди future resetAt по окнам
  - `minRemainingPercentage`
  - `hasQuotaData`
- возможно `getNearestQuotaResetAt(connectionId)`.

Это уберёт необходимость в `auth.ts`/`combo.ts` вручную копаться в `getQuotaCache().quotas`.

### 4. Интеграция стратегии в `combo.ts`
В `open-sse/services/combo.ts`:
- Добавить стратегию в комментарий supported strategies.
- Добавить branch рядом с `p2c`/`least-used`:
  - `else if (strategy === "quota-reset") { orderedTargets = await sortTargetsByQuotaReset(...); }`
- Новый helper должен:
  - сохранить исходный порядок как tie-breaker;
  - не делать сетевых fetch;
  - логировать кратко: сколько eligible, сколько skipped exhausted, какая цель первая и resetAt.
- В loop `handleComboChat()` оставить текущий `isModelAvailable` как второй защитный слой.

### 5. Добавить стратегию в типы, валидацию и UI
Файлы:
- `src/shared/constants/routingStrategies.ts`
  - добавить `"quota-reset"` в `ROUTING_STRATEGY_VALUES`.
  - добавить option в `ROUTING_STRATEGIES` с labelKey/descKey/icon.

  - расширить `ComboStrategy` или лучше синхронизировать с `RoutingStrategyValue`, если это безопасно.

  - автоматически подхватит enum после обновления constants.

- Проверить `src/lib/combos/intelligentRouting.ts`/builder, если там есть hardcoded categories стратегий.

### 6. Автообновление квоты при ошибках провайдера
Сейчас при ошибках `markAccountUnavailable()` обновляет статус/cooldown, а 429 дополнительно может маркировать quota cache через `markAccountExhaustedFrom429()` в chat handler. Нужно добавить “auto read quota” после подозрительных ошибок.

Предложение:
- Создать сервис `src/domain/quotaRefreshOnError.ts` или `src/lib/usage/quotaErrorRefresh.ts`.
- API:
  - `scheduleQuotaRefreshAfterProviderError({ connectionId, provider, status, errorText, reason })`
- Триггеры:
  - HTTP 429;
  - 402;
  - provider error text содержит `quota`, `limit`, `rate limit`, `usage_limit_reached`, `exhausted`;
  - возможно 403 только если classifier говорит quota/credits, но не ban/auth.
- Debounce/throttle:
  - per `connectionId`, например не чаще 1 раза в 60–120 секунд;
  - не блокировать пользовательский request: fire-and-forget через `setTimeout(..., 0)` или internal queue;
  - ограничить concurrency (например 2–3), чтобы не устроить thundering herd.
- Внутри вызвать `fetchAndPersistProviderLimits(connectionId, "scheduled")` или добавить source `"error"`.
  - Желательно расширить тип `SyncSource` до `"manual" | "scheduled" | "error"` для observability.
- Если live fetch успешен — quota cache обновится через `setQuotaCache()`.
- Если live fetch не поддерживается для provider/connection — тихо логировать debug/warn и не ломать request.

Точки вызова:
- `src/sse/services/auth.ts` в `markAccountUnavailable()` после классификации ошибки и перед/после DB update.
- `src/sse/handlers/chat.ts`, где уже есть `markAccountExhaustedFrom429`, можно заменить/дополнить schedule refresh, если там есть точный `connectionId`.

### 7. Обработка исчерпанных аккаунтов
Нужно сохранить два уровня защиты:
1. Новая стратегия сортирует и не ставит исчерпанные аккаунты наверх/исключает их.
2. `getProviderCredentialsWithQuotaPreflight()` всё равно проверяет quota policy перед реальным запросом.

Если quota cache говорит exhausted, но reset уже прошёл — текущий `isAccountQuotaExhausted()` уже разблокирует запись оптимистично. Для `quota-reset` нужно использовать тот же механизм, чтобы не держать аккаунт заблокированным после reset.

## Открытый вопрос перед реализацией
Название стратегии предлагаю `quota-reset`. Альтернативы: `reset-soon`, `quota-reset-priority`, `near-reset`. Если не принципиально, реализуем `quota-reset` как наиболее понятное и короткое.
